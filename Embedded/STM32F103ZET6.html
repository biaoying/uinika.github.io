基于 HAL 与 CubeIDE 的 STM32F103ZET6 开发指南

发表于 2020-10-14 分类于 Embedded 阅读次数： 本文字数： 15k 阅读时长 ≈ 13 分钟

相对于采用 LQFP48 封装的 STM32F103C8T6 微控制器，采用 LQFP144 封装的 STM32F103ZET6 时钟频率虽然同为72MHz，但提供了更加丰富的片上外设资源，例如高达512Kbytes的 Flash 程序存储器和64Kbytes的 SRAM 数据存储器，以及 12 通道DMA控制器，内置 2 个基本定时器 2 个高级定时器 4 个通用定时器，以及 3 个12bit的 ADC 与 2 个12bit的 DAC 转换器，同时还支持多达 3 组 SPI、2 组 I²C、 5 组USART、1 组CAN、1 组 SDIO 接口和 1 组全速 USB2.0 接口。

本文基于 ST 官方近年力推的 HAL（Hardware Abstraction Layer）与 LL（Low-layer）开发库，结合 STM32CubeIDE 提供的图形化配置工具，参考意法半导体官方文档《STM32F103xE Data Sheet》和《STM32F103xx Reference Manual》以及《Description of STM32F1 HAL & LL drivers》，并且结合《ARM Cortex-M3 & Cortex-M4 权威指南》一书，对于微控制器片上外设资源的编程方法进行了详细的分析与示例。

系统架构与总线

STM32F103ZET6 微控制器的系统总线架构主要由下面示意图当中的 6 个主要部分所构成：

  1. ICode 总线：用于连接 Cortex-M3 内核 的指令（Instruction）总线和 Flash 存储器 的指令接口，从而可以预先获取程序指令并进行缓存。
  2. DCode 总线：用于连接 Cortex-M3 内核 的数据（Data）总线和 Flash 存储器 的数据接口，从而实现数据加载与调试访问。
  3. System 总线：用于连接 Cortex-M3 内核 的系统总线和 总线矩阵（用于管理内核与 DMA 之间的仲裁）。
  4. DMA 总线：用于连接 DMA 的 AHB 主接口 至 总线矩阵（管理 DCode 和 DMA 对于 SRAM 与 Flash 以及其它外设的访问）。
  5. 总线矩阵：采用轮询调度算法来管理 Cortex-M3 内核 核心总线与 DMA 总线之间的访问仲裁，总线矩阵主要由 4 个主控单元（DCode 总线、System 总线、DMA1 总线、DMA2 总线）和 4 个受控单元（FLITF、SRAM、FSMC、AHB/APB 桥接器）组成。
  6. AHB/APB 桥接器：提供 AHB 与 2 条 APB 总线之间的连接，其中APB1是最高速度为36 MHz的低速外设总线，APB2则是最高速度可达72 MHz的高速外设总线。

注意：灵活静态存储控制器 FSMC 是 Flexible Static Memory Controller 的英文缩写，主要用于扩展外部的 SRAM、Nandflash、Norflash。高级高性能总线 AHB 则是 Advanced High-performance Bus 的英文缩写，主要用于连接片上各种高速外设。FLITF 是 Flash Interface 闪存接口的英文缩写，主要用于连接 Flash 存储器。

存储映射原理

存储器映射是指给芯片厂商为存储器分配地址的过程，受控单元的 Flash、SRAM、FSMC、AHB/APB 桥接器以及其它片上外设都被共同分配到一个线性的 4Gbyte 地址空间，这些可寻址的存储空间被分配为各 512MB 的 8 个块，具体映射关系请参考下面示意图以及后续的表格：

编号 描述 容量 地址范围
Block 0 分配给片上 512KB 的 Flash 程序存储器，并且存放有 Bootloader 512MB 0x0000 0000 ~ 0x1FFF FFFF
Block 1 分配给片上64KB的 SRAM 作为系统内存 512MB 0x2000 0000 ~ 0x3FFF FFFF
Block 2 分配给片上的 AHB、APB1、APB2 总线外设 512MB 0x4000 0000 ~ 0x5FFF FFFF
Block 3 分配给 FSMC 的 bank1 ~ bank2 512MB 0x6000 0000 ~ 0x7FFF FFFF
Block 4 分配给 FSMC 的 bank3 ~ bank4 512MB 0x8000 0000 ~ 0x9FFF FFFF
Block 5 分配给 FSMC 相关的寄存器 512MB 0xA000 0000 ~ 0xCFFF FFFF
Block 6 未使用，系统保留 512MB 0xD000 0000 ~ 0xDFFF FFFF
Block 7 分配给 Cortex®-M3 的内部外设 512MB 0xE000 0000 ~ 0xFFFF FFFF

最小系统设计

意法半导体 STM32F103ZETT6 分别拥有64KB的 SRAM 和512KB的 Flash、2 个基本定时器和 4 个通用定时器以及 2 个高级定时器、2 个 DMA 控制器（共 12 个通道）、3 个 SPI 接口、2 个 I²C 接口、5 个 USART 接口、1 个 USB 接口、1 个 CAN 接口、3 个 12 位 ADC 模数转换器、1 个 12 位 DAC 数模转换器、1 个 SDIO 接口、1 个 FSMC 接口、112 个 GPIO 端口。

启动模式

STM32F103ZET6 提供了 3 种不同的启动模式，这些模式可以通过BOOT0和PB2/BOOT1引脚的电平状态进行选择：

BOOT1 BOOT0 启动模式 描述
× 0 主 Flash 存储器模式 主 Flash 存储器地址被映射至启动空间0x0000 0000
0 1 系统存储器模式，即 ISP 模式 系统存储器地址被映射至启动空间0x0000 0000
1 1 内置 SRAM 模式 内置 SRAM 地址被映射至启动空间0x0000 0000

本实验电路采用 2 枚10KΩ电阻拉低了BOOT0和PB2/BOOT1引脚的电平状态，即默认是从片上的 Flash 外设地址进行启动。

时钟源

STM32F103ZET6 虽然内置有8MHz时钟源，但是出于晶振的温度漂移方面的考量，通常采用8MHz的外置无源晶振作为主时钟源。

此外，由于 STM32F103ZET6 内置的40KHz的 RTC 实时时钟源不便于精确计时，所以同样会采用一枚外置的32.768KHz时钟源。

硬件复位

STM32F103ZET6 的复位引脚NRST属于低电平有效，下面电路图中的电阻R2和电容C5组成了一个简单的 RC 复位电路。当系统上电时，C5电容两端电压为0V，此时 MCU 处于复位状态，3.3V的VDD工作电源通过电阻R2为电容C5充电，当电容C5电压升至 MCU 的高电平门槛电压时，MCU 从复位状态转换至运行状态。此外，如果在系统上电运行时，按下SW1开关，电容C5两端将会发生短路接地，从而完成手动复位。

注意：硬件电路设计时，需要注意电容和电阻元件的取值，确保NRST引脚的低电平持续时间满足复位的最小脉宽要求。

MCU 电源

为了确保供电的稳定性，STM32F103ZET6 拥有多个电源引脚，实验电路当中的每一路电源引脚都连接了0.1uf的高频滤波电容，其中VREF+/-是模数转换的参考电压，而模拟电源VDDA的参考电压通过铁氧体滤波磁珠L1接连接至3.3V电源。

开发板供电

实验电路上各种模块的工作电压主要有5V和3.3V两种，其中5V电源可以通过 2 种方法获得：

  1. 通过 USB 接口，采用电脑 USB 接口提供的5V电压，这种方式基本可以满足单个模块的负载需求。
  2. 通过 DC 接口采用外部提供的7V ~ 36V电源，由直流电源芯片LM2596S-5.0转换得到5V电源，且该芯片最大输出3A电流，驱动力较强，适用于负载模块数量较多的场合。

而3.3V电源则是利用稳压芯片AMS1117-3.3将5V电源转换获得，原理图中型号为SMAJ5.0A的瞬态电压抑制二极管D2用于避免外接5V电源不稳定时对实验电路造成损坏，同时也能在一定程度上防止由于电源反接烧坏电路。

下载调试

STLink 的 SWD 下载

STM32F103ZET6 同时支持 SWD 与 JTAG 两种调试接口，两者在调试功能上并无差异，后者是早期 ARM 的调试接口标准，需要 20 个引脚，而前者仅需 6 只引脚。当前实验电路只提供 STLink 的 SWD 调试方式支持，避免 JTAG 占用过多的 GPIO 引脚。

  • RESET引脚是硬件复位引脚，仿真器可以通过该引脚控制硬件的复位；
  • VCC_3V3电源引脚用于为 JLink 内部的电平转换芯片供电，而 STLink 的电平转换电路由于内置有3.3V上拉功能，因此不会使用到该引脚；
  • JTMS\SWDIO（PA13）：SWD 调试接口的数据线；
  • JTCK\SWCLK（PA14）：SWD 调试接口的时钟线；
  • JTDO\SWO（PB3）：用于调试的追踪功能；

ISP 串口下载

STM32F103ZET6 支持 ISP（In System Program）在系统编程方式下载程序，将BOOT0 = 1并且BOOT1 = 0系统就会进入 ISP 启动模式，通过 USB 转串口芯片 CH340G，将程序从电脑 USB 接口发送至 STM32F103ZET6 的 USART1 接口，从而将程序写入片上 的 Flash 存储器。

当DRT为高电平时三极管 Q1 导通，此时 RTS 引脚为低电平，双向开关元件 74HC1G66GW 的Z和Y引脚皆为低电平，如果 RESET 引脚为低电平 STM32F103ZET6 就会进入复位状态。这里 74HC1G66GW 主要起到在实验电路上电时，延时 RTS 引脚信号对 RESET 引脚的影响。

复位完成以后，将 RTS 和 RTS 的电平状态拉高，此时 RESET 引脚也会被拉高，这里开始将程序下载至片上 Flash，完成以后 RTS 引脚被置为高电平，BOOT0 引脚被置为低电平，系统再次复位后就会从 Flash 启动并运行下载的程序。

HAL/LL 库概览

本小节当中，将会对 HAL 和 LL 库的项目目录结构进行分析，并且列举各个库文件的具体用途：

HAL 硬件抽象层库

HAL 库对 STM32F103ZET6 的片上外设进行了比较高层次的封装，并为每个外设都提供了对应的驱动程序模块。相对于传统的标准外设库，HAL 在 STM32 系列各款微控制器上的移植性更加良好，且 API 兼容于 FreeRTOS 等嵌入式实时操作系统。总体而言，HAL 库主要由 HAL 驱动文件与用户应用文件 两个部分构成：

其中，HAL 驱动文件位于 STM32CubeIDE 所生成项目结构的Drivers\STM32F1xx_HAL_Driver目录下面，主要由如下文件组成：

文件名称 功能描述
stm32f1xx_hal_xxx.c 外设驱动文件，包含了所有 STM32 设备的通用 API，例如：stm32f1xx_hal_adc.c、stm32f1xx_hal_irda.c。
stm32f1xx_hal_xxx.h 外设驱动文件的头文件，包含有公用数据、句柄、枚举结构、定义语句和宏、以及导出的通用 API 声明，例如：stm32f1xx_hal_adc.h、stm32f1xx_hal_irda.h。
stm32f1xx_hal_xxx_ex.c 外设驱动的扩展文件，包含有某些型号特定的 API，或者新定义的 API。例如：stm32f1xx_hal_adc_ex.h、stm32f1xx_hal_dma_ex.h。
stm32f1xx_hal_xxx_ex.h 外设驱动扩展文件的头文件，包含特定的数据与枚举结构、定义语句、宏，以及导出的指定型号专有 API，例如：stm32f1xx_hal_adc_ex.h、stm32f1xx_hal_dma_ex.h。
stm32f1xx_hal.c 用于 HAL 库的初始化，包含有用于初始化 HAL 库的HAL_Init()函数，该函数将会重置所有外设，并且初始化 Flash 接口和 Systick 系统滴答定时器。
stm32f1xx_hal.h stm32f1xx_hal.c的头文件。
stm32f1xx_hal_msp_template.c 将会被拷贝至用户应用程序目录的模板文件，包含了外设 MSP 的初始与反初始化。
stm32f1xx_hal_conf_template.h 用于自定义应用程序当中驱动程序的模板文件。
stm32f1xx_hal_def.h 通用的 HAL 资源，包括通用的定义语句、枚举类型、结构体、宏定义。

文件名称 功能描述
system_stm32f1xx.c 该文件包含有系统启动时调用的SystemInit()，这里并不会涉及系统时钟配置，该工作由用户在代码中调用 HAL API 完成。
startup_stm32f1xx.s 包含有重置句柄与异常向量，对于某些工具链，甚至可以适配堆栈尺寸，以符合应用程序的需要。
stm32f1xx_flash.icf 可选，EWARM 工具链的链接文件，主要用于适配堆栈尺寸，以符合应用程序的需要。
stm32f1xx_hal_msp.c 该文件包含用户应用程序中所使用外设的 MSP 初始化/反初始化函数及其回调。
stm32f1xx_hal_conf.h 该文件允许用户去自定义当前应用程序所使用的 HAL 驱动程序，如果不进行修改，应用程序将会采用默认的配置。
stm32f1xx_it.h stm32f1xx_it.c 包含异常处理和外设中断服务程序，并会定期调用HAL_IncTick()递增stm32f1xx_hal.c中声明的timebase本地变量，默认情况下 该函数默认每间隔1ms调用 Systick 系统滴答定时器的 ISR 寄存器一次。
main.h 与 main.c 用户主程序，主要用于调用HAL_Init()、实现assert_failed()，配置系统时钟以及初始化 HAL 外设，当然更重要的作用就是放置用户业务代码。

LL 底层库

LL 提供了更加快速与轻量级的硬件底层封装，即可以独立使用，也可以与 HAL 库一起混合使用（需要分别将头文件和源文件放置在Drivers\STM32F1xx_HAL_Driver\Inc和Drivers\STM32F1xx_HAL_Driver\Src目录下面）。

文件名称 功能描述
stm32f1xx_ll_bus.h 该头文件用于核心总线控制以及外设时钟的激活与失活。例如：LL_AHB2_GRP1_EnableClock。
stm32f1xx_ll_ppp.h stm32f1xx_ll_ppp.c 用于提供LL_xxx_Init()、LL_xxx_StructInit()、LL_PPP_DeInit()之类的外设初始化函数。
stm32f1xx_ll_cortex.h ARM Cortex-M 内核相关的寄存器操作，包括系统滴答定时器LL_SYSTICK_xxxxx，低功耗模式LL_LPM_xxxxx等等。
stm32f1xx_ll_utils.h stm32f1xx_ll_utils.c 该文件包含了一些通用的 API，比如读取设备的唯一 ID 以及电子签名、时基与延迟管理、系统时钟配置。
stm32f1xx_ll_system.h 系统相关的操作，例如：LL_SYSCFG_xxx、LL_DBGMCU_xxx、LL_FLASH_xxx、LL_VREFBUF_xxx。
stm32_assert_template.h 模板文件，可以定义用于运行时检查的assert_param宏，该文件仅可用于独立使用 LL 的场景，不适用于采用 HAL 和 LL 混合开发的情况。

注意： 仅有CMSIS目录下的源码文件才会依赖于 LL 库。

HAL 和 LL 库的混合使用

虽然 STM32CubeIDE 允许混合使用 HAL 和 LL 库，但是在使用过程中依然需要注意到以下几点：

  • 避免在一个外设实例中同时使用 HAL 和 LL 的 API，否则许多 HAL 当中定义的私有字段将会相应遭到修改。
  • 包括初始化结构体在内，对于不会改变操作字段的处理过程，则可以允许同时将 HAL 和 LL 放置在同一个实例。
  • LL 库可以不受任何限制的调用任意的不以操作对象为基础的 HAL API。

除此之外，混合使用 HAL 和 LL 库时还需要注意以下几个问题：

  1. 当没有使用 HAL 的初始化或反初始化 API，且未被 LL 库的宏替换掉时，InitMsp()函数不会被调用，主堆栈指针（MSP，Main Stack Pointer）的初始化需要在用户应用程序内完成。
  2. 当流程 API 未被使用，而相应的功能由 LL 的 API 执行时，其对应的回调将不会被调用，后期处理以及错误管理应该交由用户应用程序来完成。
  3. 当 LL 的 API 被用于流程操作时，HAL 提供的 IRQ 中断请求处理函数将不会被调用，相应的 IRQ 应该由用户程序实现。每个实现了宏定义的 LL 库驱动，都需要手动读取和清除关联的中断标志位。

启动流程

STM32F103ZET6 上电运行以后就会运行startup_stm32f103zetx.s启动文件，并执行如下一系列操作：

  • 设置初始化堆栈指针 SP；
  • 设置初始化程序计数器 PC 等于 Reset_Handler；
  • 设置带有异常 ISR 地址的向量表；
  • 配置系统时钟；
  • 配置外部 SRAM 用于数据存储器（可选操作，由用户使能）；
  • 跳转至 C 库里的main，并最终调用开发人员定义的main()函数；

概而言之，STM32F103ZET6 复位之后，内部的时序逻辑电路首先将0x0800 0000地址存放的堆栈地址装入 SP 堆栈指针寄存器，然后将0x0800 0004位置存放的向量地址装入 PC 程序计数器，并从 PC 程序计数寄存器指向的物理地址取出指令进行执行，即执行复位中断服务程序Reset_Handler，该程序会调用用户 C 语言编写的SystemInit()函数来配置系统时钟以及 FSMC 总线上的外部 SRAM，然后跳转到 C 库里的main函数完成用户程序的初始化（例如变量赋初值等等），最后由其调用用户编写的main()函数开始执行用户 C 程序。

RCC 复位与时钟控制

RCC 是重置与时钟控制器（Reset and Clock Control）的缩写，主要拥有重置（Reset）、时钟（Clock）控制两方面的功能：

重置 Reset

STM32F103ZET6 主要拥有三种不同类型的重置方式：

系统重置（System Reset）：除了时钟控制器CSR寄存器里的重置标志位，以及备份域里的寄存器以外，其它所有寄存器都会被设置为复位值，系统重置主要发生在如下场景当中：

  • 复位引脚NRST处于低电平状态时，即外部复位。
  • 窗口看门狗计数条件结束，即窗口看门狗复位。
  • 独立看门狗计数条件结束，即独复位门狗复位。
  • 产生软件复位的情况下。
  • 低电压管理复位发生的时候。

电源重置（Power Reset）：当如下事件产生的时候，就会发生电源重置：

  • 发生电源开启或者电源关闭复位时。
  • 退出待机模式的时候。

备份域重置（Backup Domain Reset）：备份域仅拥有 2 种类型的重置，分别发生在如下的场景：

  • 当设置备份域控制寄存器RCC_BDCR里的BDRST位发生软件重置的时候。
  • 关闭电源 VDD 或 VBAT 然后再打开的时候。

时钟 Clock

STM32F103ZET6 的系统时钟SYSCLK可以由 HSI 时钟振荡器、HSE 时钟振荡器、PLL 锁相环三种不同的时钟源来进行驱动，并且可以选择40KHz的低速内部时钟 LSI 以及32.768KHz的低速外部时钟 LSE 作为系统的实时时钟，每个时钟源都可以按需进行独立的开关，从而优化系统的功耗特性。

通过上图提供的多种预分频器，可以对高级高速总线 AHB (最高可达72MHz)、高级外设总线 APB2 (最高可达72MHz)、高级外设总线 APB1 (最高可达36MHz) 的频率进行配置，此外 SDIO 采用的 AHB 接口的固定时钟频率为HCLK/2，而 ADC 时钟则由高速 APB2 时钟经过 2、4、6、8 分频以后获取。Cortex 系统定时器 SysTick 的外部时钟由 AHB 时钟（即HCLK）进行8分频以后获得，其它定时器的时钟频率则由硬件进行确定，这里主要存在如下 2 种情况：

  1. 如果 APB 分频器为1，那么定时器时钟频率与其所在的 APB 总线频率相同；
  2. 否则，该定时器的时钟频率将会被设置为其所在 APB 总线频率的 2 倍；

GPIO 输入输出

GPIO（通用输入输出，General Purpose Input Output）也属于一种片上外设，STM32F103ZET6 总共拥有 GPIOA、GPIOB、GPIOC、GPIOD、GPIOE、GPIOF、GPIOG 共 7 组 GPIO，每组 GPIO 包含 Px0 ~ Px15 共 16 只引脚，总计拥有 112 个 GPIO 引脚。GPIO 除了用于读取或输出数字和模拟信号以外，其对应的引脚还能够被复用为其它功能，其基本结构如下图所示：

上面示意图的上半部分对应着 GPIO 输入功能，而下半部分则对应 GPIO 的输出功能。VSS 是指公共接地，VDD 则是指电源电压。最右侧的Protection diode是保护二极管，标注为蓝色是TTL Schmitt trigger则是指施密特触发器。基于上面的电路示意图，可以将 GPIO 配置为如下 8 种工作模式：

中文名称 英文名称 功能描述
开漏输出模式 General purpose output Open-drain 用于控制 GPIO 引脚输出的高低电平状态；GPIO 输出功能由P-MOS与N-MOS管组成的反相器驱动，该模式仅会使用到N-MOS管，而不会使用P-MOS管，这里"开漏"中的"漏"字即是指 MOS 管的漏极。
推挽输出模式 General purpose output Push-pull 与开漏输出原理类似，不同点在于输出控制电路驱动当中，反相器的P-MOS管和N-MOS管都会被启用，此时 GPIO 端口即会输入电平状态，也会输出电流。
复用开漏输出模式 Alternate Function output Open-drain 以开漏输出方式，复用为其它片上外设的相应功能。
复用推挽输出模式 Alternate Function output Push-pull 以推挽输出的方式，复用为其它片上外设的相应功能。
浮空输入模式 Input Floating 该模式是 STM32 复位后的默认模式，此模式下结构图里的上下拉电阻开关都将失能。
输入上拉模式 Input Pull-down 即在浮空输入模式基础之上，使能输入电路中的上拉电阻开关。
输入下拉模式 Input Pull-up 同样在浮空输入模式基础上，使能输入电路中的下拉电阻开关。
模拟输入模式 Input Analog 进行模数转换时使用，需要配合 ADC 模数转换器外设。

除了上述 8 种工作模式以外，通过软件操作底层寄存器，还可以将 GPIO 配置为如下功能：

  • GPIO 锁定机制：通过操作配置锁定寄存器，冻结对该 GPIO 的配置操作，使其在下次复位之前，都不能再更改该 GPIO 端口的配置。
  • GPIO 复用功能的软件重映射：通过软件对寄存器的配置，将一些复用功能重新映射到其它不同的 GPIO 引脚上面，进行重映射操作时，需要开启引脚的复用功能（AFIO）时钟。

GPIO 初始化结构体

HAL 库会为 STM32F103ZET6 上的每个外设都建立初始化结构体和处理器结构体，这 2 个结构体都是定义在外设库对应的头文件当中（例如：stm32f1xx_hal_usart.h）。初始化结构体通常作为处理器结构体的一个成员变量并通过指针进行引用，而处理器结构体则会在 HAL 库当中被调用（例如：stm32f1xx_hal_usart.c），这 2 个结构体几乎囊括了对应外设的所有可选属性，因此理解它们对于掌握对应的片上外设极为有益。

但是，STM32F103ZET6 的 GPIO 外设是唯一的例外，它只拥有初始化结构体，而不存在处理器结构体，因此stm32f1xx_hal_gpio.h中定义的 GPIO 初始化结构体将会直接在 stm32f1xx_hal_gpio.c 代码中完成 GPIO 的初始化配置，其具体结构如下所示：

  • uint32_t Pin：GPIO 引脚编号，可选择 GPIO_PIN_0 ~ 15 或 GPIO_PIN_ALL，甚至使用或运算符选择多个 GPIO_PIN_0 | GPIO_PIN_4。

  • uint32_t Mode：GPIO 引脚工作模式，除上前面介绍的 8 种模式，还有另外 5 种模式可供选择：

  • uint32_t Pull：GPIO 引脚上下拉配置，

  • uint32_t Speed：GPIO 引脚最大输出速度。

GPIO 控制流程

  1. 使能 GPIO 端口时钟；
  2. 初始化 GPIO 引脚，即为 GPIO 初始化结构体成员赋值，
  3. 调用库函数 HAL_GPIO_Init() 进行初始化工作；
  4. 根据业务需求控制 GPIO 引脚高低电平状态的输出；
